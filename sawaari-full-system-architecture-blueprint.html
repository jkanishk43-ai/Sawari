PLATE 02

CLIENT LAYER

SCALE: MULTI-DEVICE
DWG NO: P02-CLI

App architecture

  • Flutter (single codebase → Android + iOS; Android ships first — Delhi commuters are ~95% Android). State: Riverpod. Local store: SQLite (drift) carrying a bundled GTFS extract, the tariff tables, and the last 30 days of trips.
  • Next.js web + PWA for desktop/search-engine surface and instant sharing of comparison links.
  • Offline-first doctrine (the Where Is My Train lesson): the app answers with zero bars. Bundled GTFS + tariffs compute bus/metro/meter fares fully on-device; app-cab estimates degrade to "last-known surge" with a stale badge; live ETAs resume on reconnect. Booking actions queue and fire when the network returns.

Maps matrix (need #3)

Layer Default pick Alternates Truth notes
Renderer MapLibre GL (BSD-3, vector, GPU) Leaflet (raster, ~42KB) for low-end devices Both open-source; zero vendor lock-in
Tiles/data Self-hosted OSM vector tiles (Planetiler build, Delhi extract) Google Maps Platform (best Delhi traffic), Azure Maps (Microsoft), Ola Maps (INR pricing, OSM-based), Mappls/MapmyIndia (deep India POIs) Self-hosted tiles cost ~₹2k/mo; Google tiles bill per-load — use only where traffic ETAs justify it
Traffic ETA Valhalla + historical speed profiles from our own GTFS-RT history Google Routes API as paid precision upgrade Ola swapped Google for in-house maps and cut a reported ~₹100 crore/yr — same playbook

Positioning stack (need #4)

  • Android: Fused Location Provider — blends GPS, NavIC (L5 band; supported on most current Snapdragon/MediaTek Indian handsets), GLONASS, cell towers, Wi-Fi. NavIC needs zero extra code; the chipset contributes it to the fix.
  • iOS: CoreLocation.
  • Offline/degraded: cell-tower coarse fix + "pick a landmark" gazetteer fallback (our 40-place Delhi gazetteer from the prototype scales to ~2,000 landmarks).

Dashboard (need #6)

Ranked comparison cards (fare, ETA, badges: CHEAPEST / FASTEST / SMART PICK / ₹0 SAHELI), expandable fare breakdowns, per-platform reliability score (cancellation odds per corridor — the 82%-cancellation wound turned into a feature), live bus positions on map, next-departure rail for stops.

Latency budget (cold path)

Stage Budget
Geocode (gazetteer/cache) 5–60 ms
Valhalla, 3 profiles parallel 40–90 ms
OTP2 itinerary plan 150–400 ms
Fare engine, all modes 5–15 ms
Ranking + BFF assembly 35 ms
Total target ≤ 1.2s P95 cold · ≤ 450ms warm

Ranking math

score = 0.55 × (fare / max_fare) + 0.45 × (eta / max_eta) — lowest score wins SMART PICK.

Weights live in config; reliability score (per-provider cancellation history on that corridor) renders beside every app-cab card.

Register properly on otd.delhi.gov.in for the realtime key (form asks org, purpose, usage type). The portal's own warning stands: static stop_times are constant-speed estimates — our ETAs come from live positions + our accumulated speed history, never from the static file.

The quote_feedback loop is the moat: users report the fare they actually paid; nightly jobs re-fit the estimate models per corridor and time-of-day; quoted-vs-actual accuracy becomes a published, marketable number.
